# 线程池执行流程详解（工厂生产线版）

> 用工厂生产线的比喻，彻底理解线程池的执行流程

---

## 一、角色定义

| 线程池概念 | 工厂比喻 | 作用 |
|-----------|---------|------|
| **corePoolSize** | 正式工 | 长期雇佣，即使没订单也养着 |
| **maximumPoolSize** | 总人数上限 | 正式工 + 临时工最多多少人 |
| **workQueue** | 仓库/订单等待区 | 订单太多时，先放这里排队 |
| **threadFactory** | 招聘标准 | 怎么招人（穿什么制服、叫什么名字） |
| **handler** | 满员处理策略 | 工厂满了，新订单怎么办 |
| **任务（Task）** | 订单 | 需要处理的工作 |
| **线程（Thread）** | 工人 | 执行任务的实体 |

## 三、分步骤详解

### 步骤1：新订单来了

```
场景：客户提交了一个新订单（execute/submit）

工厂状态检查：
- 当前正式工数量：3人
- corePoolSize：5人
- 结论：正式工还没招满，还可以招人
```

**代码对应**：
```java
executor.execute(new OrderTask());
// 或
executor.submit(new OrderTask());
```

---

### 步骤2：正式工没招满 → 招聘正式工

```
决策：当前正式工（3）< corePoolSize（5）
行动：招聘新的正式工（创建核心线程）
结果：新正式工立即执行这个订单

工厂状态变化：
- 正式工：3 → 4人
- 这个订单：立即被执行
```

**代码对应**：
```java
// ThreadPoolExecutor.execute() 内部逻辑
if (workerCountOf(c) < corePoolSize) {
    if (addWorker(command, true)) {  // true表示核心线程
        return;  // 创建成功，任务交给新线程执行
    }
}
```

**图示**：
```
提交前：                    提交后：
┌─────────────┐           ┌─────────────┐
│ 正式工：3人  │   招聘    │ 正式工：4人  │
│ 临时工：0人  │  ─────→  │ 临时工：0人  │
│ 仓库：空    │           │ 仓库：空    │
└─────────────┘           └─────────────┘
                              │
                              ▼
                         新正式工执行订单
```

```
决策：正式工已满（5人），检查是否有人空闲
发现：正式工-2号刚干完活，正在休息
行动：把订单派给正式工-2号
结果：订单立即执行

工厂状态变化：
- 正式工：5人（不变）
- 正式工-2号：空闲 → 忙碌
```

```java
// 线程池内部维护了一个workers集合
// 如果worker是空闲状态（在getTask()中等待），会被唤醒执行任务
```

### 步骤4：正式工都在忙 → 放入仓库排队

```
决策：正式工已满（5人），且都在忙
       仓库（workQueue）还有空位（容量100，当前20）
行动：订单放入仓库排队
结果：订单等待，等有工人空闲再处理

工厂状态变化：
- 正式工：5人（不变）
- 仓库：20 → 21个订单
```

**代码对应**：
```java
if (isRunning(c) && workQueue.offer(command)) {
    // 任务入队成功
    int recheck = ctl.get();
    if (!isRunning(recheck) && remove(command))
        reject(command);  // 线程池关闭了，拒绝
    else if (workerCountOf(recheck) == 0)
        addWorker(null, false);  // 至少保证有一个线程
}
```

### 步骤5：仓库满了 → 招聘临时工

```
决策：正式工已满（5人），都在忙
       仓库已满（100/100）
       当前总人数（5）< maximumPoolSize（10）
行动：招聘临时工（创建非核心线程）
结果：临时工立即执行这个订单

工厂状态变化：
- 正式工：5人（不变）
- 临时工：0 → 1人
- 仓库：100单（不变）
```

**代码对应**：
```java
else if (!addWorker(command, false)) {  // false表示非核心线程
    reject(command);  // 创建失败，拒绝
}
```

**图示**：
```
提交前：                    提交后：
┌─────────────┐           ┌─────────────┐
│ 正式工：5人  │           │ 正式工：5人  │
│ 临时工：0人  │  招聘    │ 临时工：1人  │ ← 新临时工
│             │ 临时工   │             │
│ 仓库：100单 │  ─────→  │ 仓库：100单 │
│ （已满）    │           │ （已满）    │
└─────────────┘           └─────────────┘
                              │
                              ▼
                         临时工执行订单
```

**临时工的特点**：
- 订单干完后，如果**60分钟**没新活（keepAliveTime），会被辞退
- 正式工即使没活，也不会被辞退（除非设置allowCoreThreadTimeOut）

### 步骤6：工厂满员 → 执行拒绝策略

```
决策：正式工已满（5人），都在忙
       仓库已满（100/100）
       当前总人数（10）>= maximumPoolSize（10）
行动：工厂实在处理不了，执行拒绝策略
结果：根据策略处理新订单

工厂状态变化：
- 无变化，订单被拒绝
```

**代码对应**：
```java
// 四种拒绝策略
// 1. AbortPolicy（默认）
throw new RejectedExecutionException("Task " + r.toString() + 
                                     " rejected from " + e.toString());

// 2. CallerRunsPolicy
if (!e.isShutdown()) {
    r.run();  // 调用者线程自己执行
}

// 3. DiscardPolicy
// 什么都不做，静默丢弃

// 4. DiscardOldestPolicy
if (!e.isShutdown()) {
    e.getQueue().poll();  // 丢弃最老任务
    e.execute(r);         // 尝试提交新任务
}
```

**图示**：

```
┌─────────────┐           ┌─────────────┐
│ 正式工：5人  │           │ 正式工：5人  │

│ 总人数：10人 │  ─────→  │ 总人数：10人 │
│ （已达上限） │           │ （已达上限） │
│             │           │             │
```

## 六、核心方法调用链

```
提交任务
    │
    ├── execute(Runnable command)
    │       │
    │       ├── addWorker(command, true)   // 创建核心线程
    │       │       └── Worker.run()
    │       │               └── runWorker()
    │       │                       └── task.run()  // 执行任务
    │       │
    │       ├── workQueue.offer(command)    // 任务入队
    │       │
    │       └── addWorker(command, false)   // 创建非核心线程
    │
    └── submit(Callable<T> task)
            │
            └── execute()                    // 最终调用execute
                    │
                    └── 返回FutureTask
```

## 七、面试话术（完整版）

```
【线程池执行流程】

我用工厂生产线的比喻来讲解：

线程池就像一家工厂，任务是订单，线程是工人。

当新订单来了：

第一步，看正式工招满没有（corePoolSize）。
如果没满，招聘新的正式工，让他立即执行这个订单。

第二步，如果正式工招满了，看有没有空闲的正式工。
如果有，把订单派给空闲的正式工。

第三步，如果正式工都在忙，看仓库（workQueue）还有空位吗？
如果有，把订单放入仓库排队，等工人有空再处理。

第四步，如果仓库也满了，看还能招临时工吗（maximumPoolSize）。
如果总人数还没到上限，招聘临时工执行这个订单。
临时工干完活，如果一段时间没新订单，会被辞退。

第五步，如果正式工满了、仓库满了、临时工也招满了，
工厂实在处理不了，执行拒绝策略（handler）。

拒绝策略有四种：
1. AbortPolicy：告诉客户不接了，抛异常
2. CallerRunsPolicy：老板亲自干，降低提交速度
3. DiscardPolicy：偷偷扔掉订单
4. DiscardOldestPolicy：扔掉最老的订单，接新订单

总结口诀：先核心，再队列，后非核心，最后拒绝。
```
