# synchronized锁升级过程详解

> synchronized在JDK6后进行了优化，引入了偏向锁、轻量级锁、重量级锁的升级机制

---

## 一、锁的四种状态

```
synchronized锁有4种状态，级别从低到高：

无锁 → 偏向锁 → 轻量级锁 → 重量级锁

只能升级，不能降级（偏向锁可以撤销，但不是降级）
```

| 锁状态 | 适用场景 | 优点 | 缺点 |
|--------|---------|------|------|
| **无锁** | 没有竞争 | 无开销 | 无法同步 |
| **偏向锁** | 只有一个线程访问 | 无CAS，开销最低 | 有竞争时需要撤销 |
| **轻量级锁** | 线程交替执行（低竞争） | 避免阻塞，自旋等待 | 自旋消耗CPU |
| **重量级锁** | 线程同时执行（高竞争） | 阻塞等待，不消耗CPU | 线程切换开销大 |

## 二、对象头结构

```
synchronized的锁信息存储在对象头（Mark Word）中：

┌─────────────────────────────────────────────────────────────┐
│                    64位JVM对象头结构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                  Mark Word（64位）                      │ │
│  ├───────────────────────────────────────────────────────┤ │
│  │  锁状态    │          存储内容                          │ │
│  ├───────────┼───────────────────────────────────────────┤ │
│  │  无锁      │  hashcode(31) + 分代年龄(4) + 0(1) + 01   │ │
│  ├───────────┼───────────────────────────────────────────┤ │
│  │  偏向锁    │  线程ID(54) + epoch(2) + 分代年龄(4) + 1 + 01│ │
│  ├───────────┼───────────────────────────────────────────┤ │
│  │  轻量级锁  │  指向栈中锁记录的指针(62) + 00              │ │
│  ├───────────┼───────────────────────────────────────────┤ │
│  │  重量级锁  │  指向互斥量（Monitor）的指针(62) + 10       │ │
│  ├───────────┼───────────────────────────────────────────┤ │
│  │  GC标记    │  空(62) + 11                              │ │
│  └───────────┴───────────────────────────────────────────┘ │
│                                                             │
│  最后两位是锁状态标志位：                                      │
│  01 = 无锁/偏向锁                                             │
│  00 = 轻量级锁                                                │
│  10 = 重量级锁                                                │
│  11 = GC标记                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、锁升级完整流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          synchronized锁升级流程                              │
└─────────────────────────────────────────────────────────────────────────────┘

对象创建
    │
    ▼
┌─────────────────┐
│     无锁状态     │
│  偏向锁位=0     │
│  锁标志位=01    │
└────────┬────────┘
         │ 第一个线程访问synchronized
         ▼
┌─────────────────────────────────────────────────────────────┐
│                      偏向锁（Biased Locking）                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  操作：CAS将线程ID写入对象头                                  │
│                                                             │
│  对象头：线程ID(54) + epoch(2) + 分代年龄(4) + 1(偏向锁位) + 01│
│                                                             │
│  特点：                                                      │
│  - 只有一个线程访问，无竞争                                   │
│  - 后续该线程进入同步块，只需检查线程ID是否匹配                 │
│  - 无CAS操作，开销极低（接近无锁）                            │
│                                                             │
│  适用场景：单线程反复访问同步块                                │
│                                                             │
└──────────────────────────────┬──────────────────────────────┘
                               │ 第二个线程尝试获取锁
                               │（线程ID不同）
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                    偏向锁撤销 + 轻量级锁                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  撤销偏向锁（Revoke Bias）：                                   │
│  1. 在安全点（Safepoint）暂停拥有偏向锁的线程                  │
│  2. 检查持有偏向锁的线程是否还在同步块中                       │
│  3. 如果不在，将对象头恢复为无锁状态                           │
│  4. 升级为轻量级锁                                            │
│                                                             │
│  升级为轻量级锁：                                              │
│  1. 在当前线程栈帧中创建Lock Record（锁记录）                  │
│  2. 将对象头的Mark Word拷贝到Lock Record                       │
│  3. CAS将对象头指向Lock Record（替换为00）                     │
│                                                             │
│  对象头：指向Lock Record的指针(62) + 00                        │
│                                                             │
│  Lock Record：                                               │
│  ┌─────────────────┐                                        │
│  │  displaced hdr  │ ← 原对象头的Mark Word拷贝               │
│  │  owner          │ ← 指向锁对象                            │
│  └─────────────────┘                                        │
│                                                             │
└──────────────────────────────┬──────────────────────────────┘
                               │ 多个线程竞争
                               │（CAS失败，自旋超时）
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                    轻量级锁膨胀为重量级锁                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  膨胀条件（满足其一）：                                        │
│  1. 自旋次数超过阈值（默认10次，JDK11后改为自适应）            │
│  2. 等待线程数超过CPU核数的一半                                │
│  3. 有线程在自旋时，另一个线程也尝试获取锁                      │
│                                                             │
│  膨胀过程：                                                   │
│  1. 为对象创建一个Monitor（监视器）                           │
│  2. 将Monitor地址写入对象头（替换为10）                        │
│  3. 竞争失败的线程进入Monitor的等待队列（EntryList）            │
│  4. 线程阻塞（BLOCKED状态），等待被唤醒                         │
│                                                             │
│  对象头：指向Monitor的指针(62) + 10                            │
│                                                             │
│  Monitor结构：                                                │
│  ┌─────────────────┐                                        │
│  │  Owner          │ ← 持有锁的线程                          │
│  │  EntryList      │ ← 阻塞等待的线程队列                     │
│  │  WaitSet        │ ← 调用wait()等待的线程队列               │
│  │  Recursions     │ ← 重入次数                              │
│  └─────────────────┘                                        │
│                                                             │
│  适用场景：高竞争，多线程同时执行                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、详细升级过程图解

### 1. 无锁 → 偏向锁

```
对象创建时：
┌─────────────────────────────────────┐
│  hashcode(31)                       │
│  分代年龄(4)                        │
│  偏向锁位(1) = 0                    │
│  锁标志(2) = 01  ← 无锁             │
└─────────────────────────────────────┘

Thread-1访问synchronized：
    │
    ▼ CAS操作
┌─────────────────────────────────────┐
│  线程ID(54) = Thread-1的ID          │
│  epoch(2)                           │
│  分代年龄(4)                        │
│  偏向锁位(1) = 1  ← 偏向锁          │
│  锁标志(2) = 01                     │
└─────────────────────────────────────┘

后续Thread-1再次进入：
- 检查对象头线程ID == 当前线程ID？
- 是 → 直接进入，无CAS，无开销
```

---

### 2. 偏向锁 → 轻量级锁

```
Thread-2尝试获取锁（偏向Thread-1）：
    │
    ▼ 发现线程ID不同，触发偏向锁撤销

安全点（Safepoint）操作：
1. 暂停所有线程
2. 检查Thread-1是否还在同步块中
3. Thread-1已退出 → 撤销偏向锁

升级为轻量级锁：

Thread-2的栈帧：
┌─────────────────────────────────────┐
│  Lock Record                        │
│  ├─ displaced hdr = 原对象头拷贝     │
│  └─ owner = 指向锁对象               │
└─────────────────────────────────────┘

对象头变为：
┌─────────────────────────────────────┐
│  指向Lock Record的指针(62)          │
│  锁标志(2) = 00  ← 轻量级锁          │
└─────────────────────────────────────┘
```

---

### 3. 轻量级锁 → 重量级锁

```
Thread-2和Thread-3同时竞争：
    │
    ▼ CAS操作对象头
    
Thread-2 CAS成功 → 获得轻量级锁
Thread-3 CAS失败 → 自旋等待
    │
    ▼ 自旋10次（或自适应次数）后仍失败
    ▼ 升级为重量级锁

创建Monitor：
┌─────────────────────────────────────┐
│  ObjectMonitor                      │
│  ├─ Owner = Thread-2                │
│  ├─ EntryList = [Thread-3, ...]     │
│  ├─ WaitSet = []                    │
│  └─ Recursions = 1                  │
└─────────────────────────────────────┘

对象头变为：
┌─────────────────────────────────────┐
│  指向Monitor的指针(62)              │
│  锁标志(2) = 10  ← 重量级锁          │
└─────────────────────────────────────┘

Thread-3：
- 进入EntryList队列
- 线程状态变为BLOCKED
- 等待Thread-2释放后唤醒
```

## 五、锁升级的时机和条件

| 升级方向 | 触发条件 | 具体说明 |
|---------|---------|---------|
| 无锁 → 偏向锁 | 第一个线程访问synchronized | 默认开启偏向锁（JDK6+），延迟4秒加载 |
| 偏向锁 → 轻量级锁 | 另一个线程尝试获取锁 | 偏向锁撤销需要全局安全点，有开销 |
| 轻量级锁 → 重量级锁 | 自旋失败 | 自旋次数超过阈值（默认10次）或等待线程过多 |

**批量重偏向（Bulk Rebias）**：
```
当某个类的对象被频繁撤销偏向锁（默认20次），
JVM会认为这个类不适合偏向锁，
后续该类的对象直接升级为轻量级锁，不再走偏向锁。
```

**批量撤销（Bulk Revoke）**：
```
当某个类的对象偏向锁撤销次数超过阈值（默认40次），
JVM会完全禁用该类的偏向锁，
后续该类的对象直接从无锁升级到轻量级锁。
```

## 六、JVM参数配置

```bash
# 开启/关闭偏向锁（默认开启）
-XX:+UseBiasedLocking
-XX:-UseBiasedLocking

# 偏向锁延迟加载时间（默认4秒）
-XX:BiasedLockingStartupDelay=0

# 轻量级锁自旋次数（默认10次，JDK11后自适应）
-XX:PreBlockSpin=10

# 开启/关闭轻量级锁（默认开启）
-XX:+UseLightweightLocking
```

## 八、面试话术（完整版）

### 简短版（1分钟）

```
synchronized锁有4种状态：无锁、偏向锁、轻量级锁、重量级锁。

升级过程：
1. 无锁 → 偏向锁：第一个线程访问，CAS写入线程ID
2. 偏向锁 → 轻量级锁：其他线程访问，撤销偏向锁，CAS竞争
3. 轻量级锁 → 重量级锁：自旋失败，创建Monitor，线程阻塞

只能升级，不能降级。
```

### 详细版（3-5分钟）

```
【synchronized锁升级】

JDK6之前synchronized是重量级锁，性能差。
JDK6引入了锁升级优化，有4种状态：
无锁、偏向锁、轻量级锁、重量级锁。

【升级过程】

1. 无锁 → 偏向锁
   场景：第一个线程访问synchronized
   操作：CAS将线程ID写入对象头
   特点：后续该线程进入只需检查ID，无CAS，开销极低

2. 偏向锁 → 轻量级锁
   场景：另一个线程尝试获取锁
   操作：在安全点撤销偏向锁，升级为轻量级锁
   特点：在当前线程栈创建Lock Record，CAS竞争

3. 轻量级锁 → 重量级锁
   场景：多个线程竞争，自旋超过阈值（默认10次）
   操作：创建ObjectMonitor，线程进入EntryList阻塞等待
   特点：线程切换开销大，但避免CPU空转

【对象头结构】

64位JVM对象头存储锁状态：
- 最后2位是锁标志位
- 01 = 无锁/偏向锁
- 00 = 轻量级锁
- 10 = 重量级锁

【优化参数】

-XX:+UseBiasedLocking    // 开启偏向锁
-XX:BiasedLockingStartupDelay=0  // 启动立即开启

【注意事项】

1. 锁只能升级，不能降级
2. 偏向锁撤销需要全局安全点，有开销
3. 如果确认有竞争，可以用-XX:-UseBiasedLocking关闭偏向锁
4. JDK15后偏向锁被废弃，因为性能提升不明显且复杂
```

## 九、高频面试题

### Q1：synchronized锁升级过程？

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

偏向锁：单线程无竞争，CAS写线程ID
轻量级锁：多线程交替执行，CAS自旋
重量级锁：多线程同时执行，Monitor阻塞
```

### Q2：为什么引入偏向锁？

```
场景：同一个线程反复进入同步块
优化：第一次CAS写入线程ID，后续只需检查ID
效果：无竞争时开销接近无锁
```

### Q3：轻量级锁什么时候升级为重量级锁？

```
1. 自旋次数超过阈值（默认10次）
2. 等待线程数超过CPU核数的一半
3. 有线程在自旋时，另一个线程也尝试获取锁
```

### Q4：锁能降级吗？

```
不能。锁只能升级，不能降级。

偏向锁可以撤销，但不是降级，是恢复到无锁后再升级。
```

### Q5：对象头里存了什么？

```
64位对象头：
- 无锁：hashcode + 分代年龄 + 01
- 偏向锁：线程ID + epoch + 分代年龄 + 1 + 01
- 轻量级锁：指向Lock Record的指针 + 00
- 重量级锁：指向Monitor的指针 + 10
```
