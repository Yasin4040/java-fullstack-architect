# Lite Gateway 统一错误码设计文档

## 一、评估

### 1.1 设计优点

| 维度 | 评价 | 说明 |
|------|------|------|
| **简洁性** | ⭐⭐⭐⭐⭐ | 只保留 YAML 文件 + 内置默认，去除 application.yml 配置，逻辑清晰 |
| **可维护性** | ⭐⭐⭐⭐⭐ | 错误码集中管理，修改只需改一个 YAML 文件 |
| **扩展性** | ⭐⭐⭐⭐ | 支持自定义错误码，前后端配置分离但可同步 |
| **容错性** | ⭐⭐⭐⭐⭐ | 后端不可用时前端有本地兜底配置，系统可用性高 |
| **前后端配合** | ⭐⭐⭐⭐⭐ | 通过 `/api/config/error-codes` 接口统一配置，避免硬编码 |

### 1.2 设计权衡

**做得好的地方：**
1. **配置集中化** - 专门使用 `error-codes.yml` 管理，比分散在 application.yml 中更清晰
2. **渐进式加载** - 内置默认 → YAML 文件覆盖，确保系统始终可用
3. **无 WebSocket** - 使用纯 HTTP，避免为错误码维护长连接
4. **前后端解耦** - 后端提供配置接口，前端本地兜底，各自可独立演进

**可以改进的地方（可选）：**
1. **热更新** - 当前需要重启服务才能加载新的 YAML 配置，可考虑添加 Actuator 端点实现热刷新
2. **多环境支持** - 可考虑支持 `error-codes-dev.yml`、`error-codes-prod.yml` 等多环境配置
3. **配置校验** - 启动时校验 YAML 配置格式，避免错误配置导致的问题

### 1.3 总体评价

> **这是一个实用且优雅的设计。**
> 
> 它平衡了简单性和功能性，没有过度设计。对于中小型项目来说，这个方案足够使用，且易于理解和维护。
> 
> 错误码变动频率本身就很低，不需要复杂的实时同步机制。当前的"启动时加载一次"策略完全符合业务需求。

## 二、错误码规范

### 2.1 错误码格式

采用分层编码规范，便于分类识别：

```
┌─────────────────────────────────────────────────────────────┐
│                      错误码格式                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   格式: [分类字母][四位数字]                                  │
│                                                             │
│   示例: A0201, B0001, G0001                                 │
│                                                             │
│   分类:                                                     │
│   ┌────────┬─────────────────────────────────────────────┐ │
│   │ Axxxxx │ 用户端错误 (A02xx: 认证, A03xx: 权限, A04xx: 参数)│ │
│   │ Bxxxxx │ 系统错误 (B01xx: 超时, B02xx: 限流)              │ │
│   │ Cxxxxx │ 第三方服务错误                                   │ │
│   │ Gxxxxx │ 网关特定错误                                     │ │
│   └────────┴─────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 错误码分类详解

| 分类 | 码段 | 场景 | 示例 |
|------|------|------|------|
| **A - 用户端错误** | A02xx | 认证相关 | A0201: 用户不存在, A0231: 未登录 |
| | A03xx | 权限相关 | A0301: 未授权 |
| | A04xx | 参数相关 | A0400: 参数错误, A0404: 路径不存在 |
| | A05xx | 请求服务异常 | A0501: 请求次数超限 |
| **B - 系统错误** | B00xx | 系统执行出错 | B0001: 系统执行出错 |
| | B01xx | 系统超时 | B0100: 系统执行超时 |
| | B02xx | 系统限流 | B0210: 系统限流 |
| | B03xx | 资源异常 | B0300: 系统资源异常 |
| **C - 服务错误** | C00xx | 第三方服务出错 | C0001: 调用第三方服务出错 |
| | C01xx | 中间件服务出错 | C0130: 缓存服务出错 |
| | C03xx | 数据库服务出错 | C0300: 数据库服务出错 |
| **G - 网关错误** | G00xx | 网关路由相关 | G0001: 路由不存在, G0002: 路由已存在 |
| | G01xx | 网关配置相关 | G0004: 路由配置解析失败 |

## 三、配置加载机制

### 3.1 后端加载流程

```
┌─────────────────────────────────────────────────────────────┐
│                    后端配置加载流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   应用启动                                                  │
│      │                                                      │
│      ▼                                                      │
│   ┌─────────────────┐                                       │
│   │ 加载内置默认配置 │  ← 兜底，确保系统始终可用              │
│   └────────┬────────┘                                       │
│            ▼                                                │
│   ┌─────────────────┐     存在?                             │
│   │ 检查 error-codes│ ──────────► 加载并覆盖默认配置         │
│   │    .yml 文件    │                                        │
│   └────────┬────────┘     不存在?                           │
│            │              ──────────► 使用默认配置           │
│            ▼                                                │
│   完成加载，对外提供服务                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 配置优先级

```
优先级从高到低：

1. error-codes.yml 文件中的配置（最高优先级）
2. 内置默认配置（兜底）

注意：已移除 application.yml 配置方式，保持简单
```

### 3.3 配置文件示例

```yaml
# error-codes.yml
# 此文件用于自定义错误码，会覆盖内置默认配置
# 如果不需要自定义，可以删除此文件或保持空文件

# 认证相关错误
- code: "A0201"
  message: "用户账户不存在"
  level: "warning"
  action: "none"
  showNotification: true
  duration: 3

- code: "A0231"
  message: "用户未登录"
  level: "warning"
  action: "logout"
  showNotification: true
  duration: 3

# 权限相关错误
- code: "A0301"
  message: "访问未授权"
  level: "warning"
  action: "redirect"
  redirectUrl: "/403"
  showNotification: true
  duration: 3

# 系统错误
- code: "B0001"
  message: "系统执行出错"
  level: "error"
  action: "none"
  showNotification: true
  duration: 5
  logStackTrace: true
```

### 3.4 配置字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| code | string | 是 | - | 错误码，如 "A0201" |
| message | string | 是 | - | 错误消息，显示给用户 |
| level | string | 否 | "error" | 级别: info/warning/error/fatal |
| action | string | 否 | "none" | 前端动作: none/logout/redirect/retry |
| redirectUrl | string | 否 | null | 重定向URL（action=redirect时使用）|
| showNotification | boolean | 否 | true | 是否显示通知 |
| duration | integer | 否 | 3 | 通知持续时间（秒）|
| logStackTrace | boolean | 否 | false | 是否记录堆栈（error/fatal默认true）|

```

│                    前端配置加载流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   main.ts 应用启动                                          │
```

## 七、使用示例

### 7.1 后端抛出业务异常

```java
// 使用枚举抛出
throw new BizException(ErrorCodeEnum.USER_ERROR_A0231);

// 使用自定义消息
throw new BizException("A0201", "该手机号未注册");

// 带原始异常
throw new BizException(ErrorCodeEnum.SYSTEM_ERROR_B0001, e);
```

### 7.2 前端使用错误处理

```typescript
import { useErrorHandler } from '@/hooks/useErrorHandler'

const { handleBusinessError, getErrorMessage } = useErrorHandler()

// 手动处理错误
handleBusinessError({ code: 'A0231', message: '登录过期' })

// 获取错误消息
const msg = getErrorMessage('A0201', '默认消息')
```

### 7.3 请求自动处理

```typescript
import request from '@/utils/request'

// 业务错误会自动根据配置处理
// 如 A0231 会自动登出并跳转登录页
const data = await request({ url: '/api/user/info' })
```

## 八、总结

### 8.1 设计理念

1. **约定优于配置** - 内置合理的默认值，减少配置工作量
2. **渐进式增强** - 从简单开始，按需扩展
3. **前后端分离但配合** - 各自独立，通过接口约定配合
4. **容错优先** - 确保系统在任何情况下都可用

### 8.2 适用场景

- ✅ 中小型项目
- ✅ 错误码变动不频繁的场景
- ✅ 需要统一错误处理的场景
- ✅ 多环境部署需要差异化错误消息的场景

### 8.3 扩展建议（未来可选）

1. **热刷新** - 添加 `/actuator/error-codes/refresh` 端点
2. **多环境配置** - 支持 `error-codes-{profile}.yml`
3. **配置校验** - 启动时校验 YAML 格式
4. **国际化** - 支持多语言错误消息
